# Roles Mind Map — Design Spec

**Date:** 2026-06-16
**Status:** Approved (pending implementation plan)

## Overview

Roles Mind Map — веб PWA для построения карты ролей персонажей произведения. Пользователь
заводит книги, добавляет в них персонажей и связи между ними, и видит динамический
mind-map-граф персонажей и их взаимоотношений.

Первая итерация: один локальный пользователь, запуск из одного Docker-контейнера на macOS.
Архитектура заложена под будущее размещение на VPS с множеством пользователей.

## Key Decisions

| Тема | Решение |
|------|---------|
| Семантика связи | «Я — [роль] для выбранных»: `source` является `role` для `target`. Направленное ребро. |
| Хранение | Нормализованная схема (`users`, `books`, `characters`, `relationships`). НЕ «таблица на книгу». |
| Стек | React + TS + MUI + vite-plugin-pwa; Cytoscape.js (граф); Node + Fastify + Prisma + SQLite. |
| PWA | Installable + кэш app-shell (service worker). Операции с данными — online. НЕ offline-first. |
| Multi-user | Схема и API готовы (`user_id` везде, таблица `users` с default-строкой). UI/auth — позже. |
| Подпись связей | Вариант A: роль-«чип» по центру ребра. |
| Палитра | C · Blush & Slate — пыльно-розовый primary (#c98a8a), сланцевые акценты, тёплый светлый фон (#faf6f5). |

## Architecture

Единый Docker-образ, один процесс Node:

```
┌──────────────────────── Docker image ────────────────────────┐
│  Node + Fastify (один процесс)                                │
│   ├─ /api/*        REST (JSON, валидация Zod)                  │
│   ├─ /            статика собранной PWA (React build)         │
│   └─ Prisma  ──►  SQLite  ──►  /data/app.db  (volume)         │
└───────────────────────────────────────────────────────────────┘
        ▲ браузер: React + MUI + Cytoscape.js (PWA, service worker)
```

- Fastify отдаёт и статику фронта, и API — отдельного веб-сервера нет.
- SQLite-файл на смонтированном volume → данные переживают пересоздание контейнера.
- Prisma-миграции применяются при старте контейнера.
- Задел под VPS/multi-user: в Prisma меняется только datasource SQLite→Postgres; запросы уже
  скоупятся по `user_id`; сверху добавляются auth-роуты и reverse-proxy с TLS.

## Data Model

```
users          id · name · created_at                     (сейчас 1 default-строка)
books          id · user_id→users · title · sort_order · created_at
characters     id · book_id→books · gender('male'|'female')
               · first_name(≤30) · last_name(≤30) · middle_name?(≤30)
               · age?(0..100) · pos_x? · pos_y? · created_at
relationships  id · book_id→books · source_id→characters
               · target_id→characters · role(≤30) · created_at
```

- Связь — направленное ребро: `source` является `role` для `target`.
- `pos_x/pos_y` — сохранённые координаты узла на canvas (стабильная раскладка между перезагрузками).
- Каскадное удаление: удаление персонажа удаляет все рёбра, где он `source` или `target`.

## Relationship Menu Logic

В модалке персонажа кнопка «Связи» открывает под-модалку со списком записей.
Одна запись = `Роль` (свободный текст ≤30) + `Связь` (мультивыбор других персонажей книги).

- При сохранении запись «сын → [Пётр, Жанна]» разворачивается в две строки `relationships`
  (source = текущий персонаж, role = «сын», target = Пётр / Жанна).
- Несколько записей с разными ролями → персонаж одновременно «сын» Петра и Жанны, «муж» Елены,
  «отец» Коли и Семёна и т.д.
- При редактировании существующие рёбра группируются обратно по `role` в записи.
- Сервер при сохранении делает реконсиляцию: сверяет присланный набор с текущим,
  добавляет/удаляет строки.

## Screens & Navigation (SPA, react-router)

### Экран 1 — Книги
- Первый запуск (пусто): заголовок «Roles Mind Map» по центру + кнопка «Добавить книгу».
- Нажатие → модалка с полем названия и кнопками «Отмена»/«Добавить».
- После добавления: заголовок уезжает в top bar (safe-area), список книг становится нумерованным,
  кнопка превращается в FAB «+» (правый нижний угол).
- Тап по книге → Экран 2.

### Экран 2 — Книга
- Top bar (safe-area): слева стрелка «назад», по центру «Roles Mind Map».
- Пусто: по центру название книги + кнопка «Добавить персонажа».
- Модалка персонажа: большая, safe-area, фиксированный низ с «Отмена»/«Добавить»,
  прокручиваемый контент. Поля с хинтами: Пол* (М/Ж), Имя* (≤30), Фамилия* (≤30),
  Отчество (≤30), Возраст (0–100), кнопка «Связи»* → под-модалка.
- После добавления первого персонажа экран превращается в canvas, кнопка → FAB «+» поверх canvas.

### Редактирование персонажа
- Тап по узлу → та же модалка, заполненная данными, + кнопка «Удалить» внизу
  (с подтверждением «Вы уверены?»).

## Canvas Behavior (Cytoscape.js)

- Движок cytoscape + cola (живая физика): рёбра = пружины, узлы — заряды. Это даёт «упругий
  возврат»: тянешь узел — связанные тянутся за ним, отпускаешь — раскладка мягко доводится,
  связи не «рвутся».
- Жесты: pan (перетаскивание фона), pinch/wheel-zoom, drag узла. Координаты узла сохраняются
  в `pos_x/pos_y` → стабильная карта при перезагрузке.
- Аватар = функция пол × возрастная группа (0–10 ребёнок, 11–17 подросток, 18–50 взрослый,
  51–99 старый; возраст не указан → взрослый) → один из 8 SVG-силуэтов (синий мужской /
  розовый женский). Подпись имени снизу.
- Рёбра направленные со стрелкой; роль-чип по центру линии.
- FAB «+» поверх canvas открывает модалку нового персонажа.
- Адаптивность: портрет/ландшафт; canvas занимает всё пространство под top bar.

## API (REST)

```
GET    /api/books                 список книг (sort_order)
POST   /api/books                 {title}
PATCH  /api/books/:id             {title}
DELETE /api/books/:id
GET    /api/books/:id/graph       {nodes:[characters], edges:[relationships]}
POST   /api/characters            {book_id, gender, ...поля, relations:[{role,targetIds[]}]}
PATCH  /api/characters/:id        те же поля + relations (сервер реконсилит рёбра)
PATCH  /api/characters/:id/pos    {pos_x,pos_y}   (дешёвое сохранение позиции)
DELETE /api/characters/:id        (каскад по рёбрам)
```

- Валидация Zod на сервере, та же логика лимитов на клиенте (≤30, 0–100, обязательные поля).
- Все запросы скоупятся по текущему `user_id` (сейчас default).

## Docker

- Multi-stage: stage 1 — `vite build` фронта; stage 2 — сборка сервера (TS→JS);
  финал — node-образ со статикой + сервером.
- `docker compose`: маппинг порта + volume на `/data` (SQLite). Один сервис, один процесс.
- Локальный запуск на macOS; та же конфигурация переносима на VPS.

## Testing

- **Backend (Vitest + supertest):** CRUD книг/персонажей; отдельно — юнит-тесты реконсиляции
  связей (добавление/удаление рёбер, разворачивание записи в строки, каскад при удалении).
- **Frontend (Vitest + React Testing Library):** валидация форм (лимиты, обязательность),
  логика под-модалки связей, чистая функция-адаптер `модель → элементы Cytoscape`,
  выбор аватара по полу/возрасту.
- **E2E (Playwright, happy-path):** создать книгу → создать персонажа → увидеть узел на canvas.

## UI Style

- Material Design (MUI), пастельные светлые тона, без кричащих цветов.
- Палитра C · Blush & Slate: primary #c98a8a, top bar #dcb6b6, фон #faf6f5,
  сланцевые акценты для рёбер/второстепенного. Мужские аватары — приглушённо-синие,
  женские — розовые.
- Адаптив: портретная и ландшафтная ориентации; top bar и модалки уважают safe-area.

## Out of Scope (первая итерация)

- Аутентификация и регистрация (схема готова, UI позже).
- Offline-first синхронизация (только кэш app-shell).
- Postgres (переход — смена Prisma datasource при выходе на VPS).
