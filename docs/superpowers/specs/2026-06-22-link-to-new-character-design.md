# Связь с новым персонажем

**Дата:** 2026-06-22

## Проблема

Сейчас связь можно создать только между **существующими** персонажами:
`RelationsModal` (открыт из `CharacterModal`) показывает меню уже созданных
персонажей, а связь стейджится локально и сверяется сервером по `otherId` —
то есть второй конец связи обязан существовать.

Если нужного персонажа ещё нет, пользователь вынужден прерваться, отдельно
создать его, затем вернуться и связать. Цель — позволить создать нового
персонажа **прямо из потока добавления связи**, сразу связав его с текущим.

## Поток (UX)

Внутри `RelationsModal`, открытого из настроек персонажа **A**:

1. Кнопка **«+ Добавить связь»** показывается **всегда** (раньше — только при
   наличии доступных персонажей), потому что создать нового можно в любой
   момент, даже когда A — первый персонаж книги.
2. Клик по «+ Добавить связь» открывает **диалог-развилку** `LinkChoiceDialog`:
   заголовок «Создать нового персонажа или связать с существующим?», кнопки
   **«Существующий»** и **«Новый персонаж»**.
   - **«Существующий»** → текущее меню выбора персонажей (поведение без
     изменений). Пункт недоступен, если связывать не с кем (нет других
     персонажей).
   - **«Новый персонаж»** → запускается передача управления (см. ниже).
3. **«Новый персонаж»**: текущие staged-строки связей сохраняются в A → A
   **валидируется и сохраняется** (create/update + аватар) → окно A
   закрывается → открывается чистая форма **создания B**, в которую уже
   «зашита» связь с A.
4. Пользователь заполняет B, жмёт «Добавить» → B создаётся со связью
   `{ otherId: A.id, role: "", color: null }` → возвращаемся на карту, A и B
   связаны.

Связь A↔B создаётся с **пустой ролью и цветом по умолчанию** (обычная
безымянная линия). Роль/цвет можно задать позже через «Связи» любого из
персонажей.

### Жизненный цикл окна A (принятое решение)

При выборе «Новый персонаж» A **сохраняется немедленно**, и его окно
закрывается; на экране остаётся только форма создания B. Это:

- проще и надёжнее в текущей архитектуре (каждое окно сохраняет само себя,
  кросс-модальной оркестрации состояния нет);
- сохраняет правки A;
- работает, даже если A ещё не создан (create-режим).

Компромисс: если отменить создание B, A уже сохранён — но без новой связи.
Это осознанно принято.

## Изменения по компонентам

Каждое окно персонажа остаётся самодостаточным: оно сохраняет **само себя**, а
связь с A передаётся в B как заранее заданная строка связи.

### Новый компонент `LinkChoiceDialog`

По образцу `ConfirmDialog`. Простой MUI `Dialog`: заголовок-вопрос + две кнопки.

- Пропсы: `open`, `canUseExisting: boolean` (для disable «Существующий»),
  `onExisting`, `onCreateNew`, `onCancel`.
- Обёрнут в `useBackClose`.

### `RelationsModal`

- Кнопка «+ Добавить связь» рендерится **всегда**; по клику открывает
  `LinkChoiceDialog` вместо прямого меню персонажей.
- «Существующий» → открывает существующий `Menu` (как сейчас). Если
  `available.length === 0` — пункт недоступен.
- «Новый персонаж» → вызывает новый пропс `onCreateNew(rows)`, **передавая
  текущие staged-строки** наверх (чтобы они не потерялись при сохранении A).
- `useBackClose` для диалога-развилки.

### `CharacterModal`

- Новый пропс `onCreateLinked?: (input, avatar) => void`, проксируется в
  `RelationsModal` как `onCreateNew`.
- Обработчик: принять `rows` из `RelationsModal` → положить их в локальный
  `relations` → закрыть `RelationsModal` → провалидировать форму (как обычный
  `submit()`) → при успехе вызвать `onCreateLinked(input, avatar)` вместо
  `onSubmit`. При ошибке валидации — показать ошибки, остаться в окне A.

### `BookScreen`

- Расширить состояние модалки: для create-режима добавить опциональный
  `linkedTo?: string` (id персонажа A).
- Новый `submitAndCreateLinked(input, avatar)`: сохраняет A (create/update +
  аватар, как в `submit`), `await refresh()` (чтобы A появился в
  `graph.nodes`), затем `setModal({ mode: "create", linkedTo: aId })`.
- Сборка `initial` для linked-create: дефолтные поля + `relations:
  [{ otherId: linkedTo, role: "", color: null }]`. `others` для B = все узлы
  (включая только что сохранённого A).
- `onCreateLinked` прокидывается в `CharacterModal` → `submitAndCreateLinked`.

### Поток данных по связи

B создаётся через `createCharacter`; серверный `reconcileRelationships` для B
видит `{ otherId: A.id }` и создаёт каноническую дугу A↔B. Перезагрузка графа
показывает линию. A повторно не трогаем — его reconcile уже отработал при
сохранении и связь не удалит (она добавлена со стороны B).

## Граничные случаи

- **Рекурсия B → C.** Форма создания B — обычный `CharacterModal` с тем же
  `onCreateLinked`. Из B можно так же создать C, связанного с B. Механизм
  один на каждом уровне (сохранить текущего → открыть следующего), стек не
  растёт — работает без спец-обработки.
- **A ещё не сохранён (create-режим).** Поток одинаков для edit и create: при
  «Новый персонаж» A сначала валидируется и создаётся (получает id), затем
  открывается B. Если обязательные поля A не заполнены — показываем ошибки и
  **остаёмся в окне A**, B не открывается.
- **Нет других персонажей.** Кнопка «+ Добавить связь» видна; в развилке
  «Существующий» недоступна, «Новый персонаж» работает. Основной сценарий для
  первой пары персонажей книги.
- **Кнопка Back / системная «назад».** `LinkChoiceDialog` подключён через
  `useBackClose` и закрывается как верхний слой. После перехода к B окно A уже
  закрыто, поэтому Back на форме B закрывает B и возвращает на карту (A уже
  сохранён).
- **Отмена создания B.** A уже сохранён при переходе; отмена B оставляет A
  сохранённым, но без новой связи (осознанный компромисс).
- **Дубли связей.** Если в форме B вручную добавить связь обратно к A через
  «Существующий» — это та же пара; `@@unique([sourceId, targetId])` + reconcile
  по `otherId` не создадут второй дуги. Безопасно.

## Тестирование

- **`RelationsModal`** — развилка: «Существующий» → меню; «Новый персонаж» →
  колбэк с текущими строками; кнопка «+ Добавить связь» видна при пустом списке.
- **`LinkChoiceDialog`** — disable «Существующий» при `canUseExisting=false`.
- **`BookScreen`-поток** — сохранение A → открытие B с preset-связью →
  создание B → дуга A↔B в графе.
- **E2e** — по желанию.

## Вне рамок (YAGNI)

- Запрос роли/цвета связи в этом же потоке (создаём с пустой ролью, цветом по
  умолчанию; редактируется позже).
- Какой-либо серверный API за пределами существующих `createCharacter` /
  `updateCharacter` (связь едет в полезной нагрузке B, reconcile без изменений).
