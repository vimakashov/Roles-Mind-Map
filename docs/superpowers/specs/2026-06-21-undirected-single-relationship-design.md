# Доработка: одна ненаправленная связь на пару персонажей

**Дата:** 2026-06-21
**Статус:** дизайн утверждён, готов к плану реализации

## Постановка задачи

Между двумя персонажами может существовать **только одна связь**, граф —
**ненаправленный**. У персонажа может быть много линий к другим персонажам, но
между конкретной парой персонажей не может быть более одной связи. Нужно убрать
возможность добавить связь с персонажем, с которым связь **уже есть** (в любом
направлении).

## Текущее состояние (что меняем)

- Связь **направленная**: строка `Relationship` означает «source — [role] для
  target». На холсте рисуется стрелка (`target-arrow-shape: triangle`).
- Уникальность: `@@unique([sourceId, targetId, role])` — допускает одновременно
  `A→B "друг"`, `B→A "враг"` и `A→B "брат"` между одной парой.
- Карточка персонажа (`groupEdges`) показывает только **исходящие** связи; связь,
  созданную «с другой стороны», текущий персонаж не видит.
- Карточка «Связи» сгруппирована по роли: одна роль → мультивыбор персонажей.

## Принятые решения

1. **Полностью ненаправленный граф** — одна строка-связь на неупорядоченную
   пару; на холсте линия без стрелки; в карточке видны все связи персонажа,
   включая созданные «с другой стороны»; любую можно редактировать/удалять.
2. **Роль — симметричная метка** (описание связи без направления, напр. «друзья»,
   «семья»), одинаково читается с обеих сторон.
3. **Карточка «строки»** — один персонаж = одна связь (строка: персонаж + роль +
   цвет + удалить).
4. **Гарантия уникальности в БД через канонический порядок** хранения
   (`sourceId < targetId`).

## Дизайн

### 1. Модель данных (`server/prisma/schema.prisma`)

- `Relationship` представляет ненаправленную связь, **одна строка на
  неупорядоченную пару**.
- Колонки `sourceId`/`targetId` сохраняются, но хранятся **канонически**:
  `sourceId` = лексикографически меньший id, `targetId` = больший. Направление
  больше не несёт смысла — это просто два конца связи.
- `@@unique([sourceId, targetId, role])` → **`@@unique([sourceId, targetId])`**.
  В каноническом порядке это гарантирует ровно одну строку на пару на уровне
  SQLite.
- `role` остаётся `String` — симметричная метка, по-прежнему допускает пустую
  строку `""`, ограничение длины 30. `color` без изменений (nullable, `null` =
  дефолтный `EDGE_COLOR`).

### 2. Миграция существующих данных

Формальных миграций нет — `server/src/server.ts` выполняет `prisma db push` на
старте. Если в существующей БД есть дубли пары (`A→B` и `B→A`, либо несколько
ролей на пару), новый unique-индекс не создастся и push упадёт. Поэтому **до**
`db push` добавляется идемпотентный шаг `normalizeRelationships()` (raw SQL, под
guard'ом «таблица `Relationship` существует»):

- для каждой неупорядоченной пары оставляем **самую раннюю** строку по
  `createdAt`, остальные удаляем (детерминированное правило; на личном
  инструменте потеря лишней связи приемлема);
- оставшиеся строки приводим к каноническому порядку (меняем местами
  `sourceId`/`targetId`, если `sourceId > targetId`).

На свежей БД таблицы ещё нет → шаг пропускается, `db push` создаёт таблицу сразу
с новым unique. Порядок на старте: `normalizeRelationships()` → `prisma db push`.

### 3. Сервер

- **Wire-формат** меняется с группировки по роли на плоский список соединений:

  ```ts
  relations: Array<{ otherId: string; role: string; color: string | null }>
  ```

  `role` проходит тот же валидатор `name30` (trim, max 30, default `""`).
  `otherId` — id персонажа на другом конце связи.

- **`reconcileRelationships(tx, bookId, characterId, connections)`** становится
  «по инцидентным рёбрам»:
  - `desired` = `Map<otherId, { role, color }>`;
  - `existing` = строки, где `sourceId == characterId` **ИЛИ**
    `targetId == characterId` (обе стороны); для каждой `otherId` =
    противоположный конец;
  - удаляем инцидентные строки, чей `otherId` не в `desired`;
  - создаём недостающие связи **в каноническом порядке** (`min/max` пары);
  - обновляем `role`/`color` у изменившихся существующих строк.
  - Так как смотрим обе стороны, связь, созданная «с другой стороны», видна и не
    дублируется.
  - Self-target (`otherId == characterId`) игнорируется.

- `getBookGraph` — формат рёбер без изменений (`id, sourceId, targetId, role,
  color`); фронт трактует их как ненаправленные.

### 4. Веб

- `web/src/types.ts`: `RelationEntry`/`RelationTarget` → плоский
  `RelationConnection = { otherId: string; role: string; color: string | null }`;
  `CharacterInput.relations: RelationConnection[]`.
- `web/src/lib/relations.ts`:
  - `groupEdges` → **`incidentConnections(characterId, edges)`** — собирает все
    рёбра, где персонаж является любым из концов, `otherId` = противоположный
    конец (так в карточке видны и «входящие» связи). Порядок — по `createdAt`
    рёбер (рёбра уже отсортированы).
  - `expandEntries` упрощается / удаляется — список уже плоский и совпадает с
    wire-форматом.
- **`web/src/components/RelationsModal.tsx`** переписывается на строки: каждая
  связь = персонаж (read-only) + поле «Роль» (необязательно, max 30) + цветной
  свотч (существующий `Popper` + `@uiw/react-color`, оставить `Popper`, не
  `Popover`) + кнопка удалить. «+ Добавить связь» = одиночный выбор персонажа из
  тех, с кем связи ещё **нет**, и не себя.
- **`web/src/canvas/MindMap.tsx`**: стиль ребра `target-arrow-shape: "triangle"`
  → `"none"` (линия без стрелки). `line-color`/`label`/цвет — как есть.
- `web/src/components/CharacterModal.tsx`: тип состояния `relations`
  обновляется; счётчик «Связи (N)» = число соединений.

### 5. Тройная гарантия ограничения

- **БД:** `@@unique([sourceId, targetId])` при каноническом порядке — максимум
  одна строка на пару.
- **Код:** `reconcileRelationships` канонизирует и смотрит обе стороны — вторую
  строку не создаёт.
- **UI:** в выпадающем списке «Добавить связь» скрыты уже связанные персонажи —
  попытка дубля исключена.

## Вне области задачи (YAGNI)

- Сохранение направленных ролей / отдельной семантики «кто кому кем приходится».
- Множественные связи/роли между одной парой.
- Ручной выбор, какую из дублирующихся связей оставить при миграции (правило
  фиксированное: самая ранняя по `createdAt`).
