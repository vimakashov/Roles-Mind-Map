# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Roles-Mind-Map — a simple mind map for book characters (per `README.md`). Apache 2.0 licensed.

## Status

Implementation is complete and runs via Docker (all phases committed; CRUD for books and characters, mind-map canvas, custom character avatars, undirected single relationship per character pair (arrowless lines), per-edge relationship line colours, per-character free-form comments, create-and-link a brand-new character from the relations flow, edit/delete a single relationship by tapping its line on the canvas, the open book's title shown in the canvas top bar, connection-based avatar/name and edge-length scaling with label-aware overlap avoidance, the canvas auto-fitted to show all characters on load, system Back button closes the top-most modal, public read-only book sharing via a copyable `/share/:bookId` link, PWA).

### Architecture

npm-workspaces monorepo with two packages:

- **`server/`** — Fastify 4 REST API, Prisma 5 ORM, SQLite database. Builds to `server/dist/`. In production, also serves the compiled web bundle as static files.
- **`web/`** — React 18 + TypeScript + MUI + Cytoscape.js PWA (via `vite-plugin-pwa`). Built with Vite.

**Single Docker image** (multi-stage): builds both packages, runs the Fastify server which serves the web bundle.

**Schema** (normalized SQLite): `User`, `Book`, `Character`, `Relationship`, `CharacterAvatar`. `Character.firstName` is required; `lastName` and `middleName` are both nullable (`String?`, stored as `null` when blank). A `Relationship` row is an **undirected** edge: exactly **one row per unordered pair** of characters. `sourceId`/`targetId` are stored **canonically** (`sourceId` = lexicographically smaller id, `targetId` = larger) — direction carries no meaning, they are just the two ends. Uniqueness is enforced by `@@unique([sourceId, targetId])`. `role` is a **symmetric label** (e.g. "друзья", "семья"), `NOT NULL` but **accepts an empty string `""`** (an unlabelled line, max 30), and carries an optional `color` (hex `#rrggbb`, nullable) for its canvas line; `null` renders with the default `EDGE_COLOR`. `CharacterAvatar` is a 1:1 with `Character` (shared PK `characterId`, `onDelete: Cascade`) holding the avatar bytes (`data Bytes`, `mimeType`, `width`, `height`, `updatedAt`) in a separate table so the graph query never loads blobs. `Character.deceased` (`Boolean @default(false)`) is a symmetric status flag rendered as an X+veil overlay on the character avatar in both the modal and the canvas. `Comment` is a 1:N child of `Character` (`onDelete: Cascade`) holding free-form notes (`text`, max 2000). Comments are **staged** in `CharacterModal` and reconciled on character save (like relations), never persisted independently; the graph payload embeds each character's comments (`{ id, text }`, ordered by `createdAt`).

**Book rename** — a pencil icon in `TopBar` (`onEdit`, left of the trash) opens a rename dialog in `BookScreen` that PATCHes `PUT`-style `/api/books/:id` via `api.updateBook(id, title)` and refreshes. The dialog reads the current title from the graph payload (`getBookGraph` returns `{ title, nodes, edges }`; `BookGraph.title` is `title?: string` on the web side — optional only so `tsc` keeps compiling the test fixtures that omit it, the real payload always sets it).

**Book title in the top bar** — `TopBar` takes an optional `title?: string` prop and renders `title || "Roles Mind Map"` (single-line ellipsis truncation via `noWrap` + `overflow: hidden` + `textOverflow: ellipsis`, bounded by the existing `flex: 1` between the two fixed 96px icon boxes). `BookScreen` passes `title={graph.title}`; the `|| "Roles Mind Map"` fallback covers the loading flash (`graph.title` is `""` until the fetch resolves) and `BooksScreen` (renders `<TopBar />` with no title, so it keeps the default). After a rename, `BookScreen` already re-fetches the graph, so the bar updates automatically. No API/schema change — the title already arrives via `getBookGraph`.

Data persists in the `rmm-data` Docker volume.

**Character avatars** — each character shows either a *custom uploaded image* or a *schematic silhouette*.

- **Schematic (default)** — `web/src/lib/avatarSvg.ts` exposes `avatarSvgMarkup(gender, age)`, the single source of truth for the silhouette (gender colour + age-based head radius). It is consumed two ways from the same call: the React `Avatar` component renders it via `dangerouslySetInnerHTML`, and `graphAdapter.ts` emits it as a `data:image/svg+xml,` URI (`avatarUri`) used as the Cytoscape node `background-image`. Don't re-inline the SVG geometry anywhere else.
- **Custom (uploaded)** — no server-side image processing (pure-JS server preserved). The browser validates the file (`web/src/lib/avatarImage.ts`: type/size, then pixel dims via `loadImage`), the user circular-crops it (`AvatarCropDialog` + `react-easy-crop`), and `bakeToWebp` renders the crop to a **512×512 WebP** blob. Only that small blob is uploaded as base64 JSON via `api.setAvatar` to `PUT /api/characters/:id/avatar` (validated by `avatarUploadSchema`; mime literal `image/webp`, max dim, 2 MB decoded byte cap). Bytes are served from `GET /api/characters/:id/avatar` (used as both `<img src>` and the Cytoscape `background-image`) and removed via `DELETE`. The graph payload exposes only `avatarUpdatedAt` (never bytes); when set, `Avatar`/`graphAdapter.ts` build the GET URL with a `?v=<avatarUpdatedAt>` cache-bust (`api.avatarUrl`) instead of the schematic data URI. `CharacterModal` stages the change (`AvatarChange` = none/set/remove) locally and `BookScreen.submit` reconciles it after the character save (cancel discards the staging).

**Back-button modal handling** — the browser/system Back button closes the top-most open overlay (peeling one layer per press) instead of navigating the router. A singleton History-API manager (`web/src/lib/backStack.ts`) keeps a stack of open overlays plus a count of throwaway history "sentinels"; the thin `useBackClose(open, onClose)` hook (`web/src/lib/useBackClose.ts`) wires each overlay in. Wired overlays: `ConfirmDialog`, `AvatarCropDialog` (ignores Back while baking), `CharacterModal` + its avatar `Menu`, `RelationsModal` + its colour `Popper`, `RelationEditModal` + its colour `Popper`, `BooksScreen` new-book dialog, `BookScreen` rename + delete-book dialogs. See the Gotchas entry for the same-URL sentinel / `guardedPops` mechanics.

**Character comments** — a «Комментарии (N)» button in `CharacterModal` opens `CommentsModal` (list, mirrors `RelationsModal`): empty state shows «Добавить комментарий +»; each row is `«{i+1}. {first 15 chars}…»` with a trash button; tapping a row opens `CommentEditDialog` (multiline, max 2000) to edit. Edits are staged locally (`CommentItem = { id: string | null; text }`, `id` null = new) and ride the character create/update body; `reconcileComments` (`server/src/services/comments.ts`) creates null-id rows, updates matching ids, deletes the rest — scoped to the character.

**Link to a new character** — in `RelationsModal`, the «+ Добавить связь» button (now always shown, even when there is no one left to link to) opens `LinkChoiceDialog` («Создать нового персонажа или связать с существующим?»). «Существующий» opens the usual character `Menu` (disabled in the choice when `available.length === 0`); «Новый персонаж» calls `onCreateNew(rows)` with the current staged rows. `CharacterModal` commits those rows, validates, and calls `onCreateLinked(input, avatar)` instead of `onSubmit`. `BookScreen.submitAndCreateLinked` saves character A immediately (via shared `saveCharacter`), refreshes, then opens a fresh create modal with `linkedTo = A.id`; the modal is seeded by the new `presetRelations` prop (so `gender` starts empty), and a `key={mode}:{id|linkedTo|new}` forces a remount on the A→B transition so the form resets. B's own save creates the canonical A↔B edge via the existing reconcile (no new server API; relations wire shape unchanged). Cancelling B leaves A saved without the link — an accepted trade-off of saving A up front.

**Edit a relationship by tapping its line** — two dedicated REST endpoints persist a single edge by id, isolated from the character-save reconcile: `PATCH /api/relationships/:id` (body validated by `relationUpdateSchema` = `{ role: string trim/max30/default "", color: hex-or-null }`, returns the updated row) and `DELETE /api/relationships/:id` (204). A missing id throws Prisma `P2025`, which the global error handler in `app.ts` maps to `404` — no explicit not-found check (matches the character DELETE route). Both live in `server/src/routes/relationships.ts`, registered after `characterRoutes`. On the canvas, `MindMap` gains an optional `onEdgeTap?: (id) => void` (bound via the `onEdgeTapRef` ref like the node handlers, so it survives attribute-only re-syncs); `BookScreen` resolves the tapped id to a `Relationship` from `graph.edges` and renders `RelationEditModal` (a standalone `Dialog` mirroring one `RelationsModal` row: shows «{sourceName} — {targetName}», a role field, a colour `Popper`, and a trash button → `ConfirmDialog`). «Сохранить» calls `api.updateRelation(id, { role: role.trim(), color })`; the trash confirms then `api.deleteRelation(id)`; both fire `onChanged` which closes the modal and re-fetches the graph. `relationUpdateSchema` is a **full replace** — both `role` and `color` are always written, so a partial PATCH sending one field would clobber the other; harmless as wired because the modal always sends both from local state.

**Connection-based canvas scaling** — a character's avatar+name and the lengths of its relationship lines scale with its *degree* (number of incident edges). Pure helpers in `web/src/lib/layout.ts`: `scaleForDegree(d) = 1 + SCALE_PER_EDGE(0.5)·d` is **uncapped** (a well-connected character keeps growing) and drives node `width`/`height` (`BASE_NODE_SIZE 46 × scale`) and name `font-size` (`BASE_FONT_SIZE 11 × scale`); `edgeScaleForDegree(d) = 1 + EDGE_SCALE_PER_EDGE(0.1)·d` grows edge length far more gently than the avatar. `graphAdapter.toElements` counts degree from `graph.edges` (both endpoints) and emits `data.scale` + `data.edgeScale` per node (degree 0 → 1.0). `MindMap`'s cola `edgeLength` is `edgeLengthForNodes(scaleA, scaleB, edgeScaleA, edgeScaleB) = max(edgeLengthForScales(edgeScaleA, edgeScaleB), BASE_NODE_SIZE·(scaleA+scaleB) + EDGE_CLEARANCE(120))` — small nodes keep the gentle 0.1 growth, but past ~5 connections the geometric floor lifts a big node's links so its avatar/label clears its neighbours. The **hard** no-overlap guarantee is cola's overlap avoidance with `nodeDimensionsIncludeLabels: true` (see the Gotchas entry), not the edge length. No server/schema/API change; purely visual. The in-place sync effect spreads the new `scale`/`edgeScale` like any other mutable `data` field, and add/remove of a relationship is an id-set change that re-inits + re-runs cola, so both paths recompute scales for free.

**Fit all characters on load** — on mount `MindMap` frames the whole graph: `cy.fit(undefined, FIT_PADDING(50))` zooms so every element's bounding box (labels included) fits the viewport, so screen orientation/aspect ratio are respected and it zooms out as far as needed. The cola layout keeps `fit: false` (cola's built-in `fit` refits on *every* tick and would hijack pan/zoom — see Gotchas). The initial fit is **synchronous** (frames saved maps with stored positions immediately); while a fresh layout spreads nodes out, a rAF-debounced refit on node `position` events keeps the graph framed, released on the user's first `scrollzoom`/`pinchzoom`/`tapstart` gesture or after `FIT_SETTLE_MS(4000)`. The effect cleanup clears the timer and cancels the pending rAF (and `cy.destroy()` stops cola).

**Auth (nickname + password, cookie session, no logout)** — the SPA is gated behind login. `web/src/AuthGate.tsx` calls `api.me()` on mount and renders `AuthScreen` (`web/src/screens/AuthScreen.tsx`: register/login mode toggle, inline zod errors via `nicknameField`/`passwordField`, a «Забыли пароль?» modal pointing at `https://mkv.qa/`) when anonymous, the router when authenticated. On the server, `@fastify/cookie` is registered in `app.ts` with `getSessionSecret()` (env `SESSION_SECRET`, else a generated secret persisted next to the SQLite DB); a `preHandler` resolves the signed `rmm_session` cookie (value = userId, `httpOnly`, `sameSite: lax`, 10-year `maxAge`) into `req.user` and 401s any `/api/*` request except `/api/auth/*`. `server/src/routes/auth.ts` exposes `POST /api/auth/register` (201 + cookie, 409 on duplicate), `POST /api/auth/login` (200 + cookie, 401 on bad creds), `GET /api/auth/me`. Passwords are hashed with `node:crypto` `scrypt` (`server/src/auth.ts`, `"<salt>:<hash>"`, no native deps). Nicknames are unique/looked-up **case-insensitively** via raw `LOWER(name)` (Prisma SQLite lacks `mode: "insensitive"`); stored as entered. All data routes scope to `req.user.id` via ownership helpers in `server/src/ownership.ts` (cross-user id → 404). `ensureAdminUser` (`server/src/adminUser.ts`, replaces the old `ensureDefaultUser`) seeds `synthmadness/6629` on boot, upgrading the legacy `default-user` row in place so its existing books carry over; `DEFAULT_USER_ID` is re-exported for the service-direct tests.

**Public read-only book sharing** — a book owner copies a permanent public link `<origin>/share/<bookId>` that opens a login-free, read-only view of that book's mind-map. The link is **derived from `Book.id`** (no schema column, token, or migration — every book is implicitly shareable). On the server, `server/src/routes/share.ts` (`shareRoutes`, registered after `relationshipRoutes`) exposes two **unauthenticated** GETs, and `app.ts`'s gate exempts `/api/share/` (the `isShare` branch alongside `isAuth`): `GET /api/share/:bookId/graph` returns the same `getBookGraph` payload (`404` if the book is missing) and `GET /api/share/:bookId/characters/:characterId/avatar` returns avatar bytes (book-scoped: `404` if the character isn't in that book or has no avatar; same `immutable` cache header as the authed route). These are **read-only** — no public write endpoints exist. On the web, the `/share/:bookId` route lives **outside `AuthGate`** in `App.tsx` (`BrowserRouter` now wraps the `Routes`; the two authed routes each wrap their screen in `<AuthGate>`), so it never calls `api.me()` or shows the login screen. `ShareScreen` (`web/src/screens/ShareScreen.tsx`) fetches via `api.getSharedGraph`, reuses `MindMap` with the new **`avatarUrl` builder prop** (threaded through `toElements(graph, { avatarUrl })` — defaults to `api.avatarUrl`, here `api.sharedAvatarUrl` for the book-scoped public route), renders a read-only `CharacterView` card (`web/src/components/CharacterView.tsx` — plain-text fields/relations/comments, no edit controls) on node tap, and shows «Ссылка недействительна» on fetch error / «Персонажей пока нет» when empty. `TopBar` gains an optional `onShare?` prop (a `ShareIcon`, aria-label «поделиться», left of the edit pencil, shown only when provided); `ShareScreen`'s `TopBar` gets **no** handler props so all icons (back/share/edit/delete) are hidden and there's no add FAB. The owner's `BookScreen` wires `onShare` → `copyToClipboard` (see Gotchas) of `${window.location.origin}/share/${bookId}` + a `Snackbar` toast.

### Gotchas (learned the hard way — keep in mind)

- **Clipboard copy must not rely solely on `navigator.clipboard`** — Safari/WebKit is stricter than Chrome about the async Clipboard API (absent in non-secure HTTP contexts, flaky on iOS / older macOS Safari), so the share-link copy goes through `copyToClipboard` (`web/src/lib/clipboard.ts`): it tries `navigator.clipboard.writeText` then falls back to a synchronous hidden-`<textarea>` + `document.execCommand("copy")` (uses `setSelectionRange` so iOS Safari selects the text). It returns a **boolean** and `BookScreen` surfaces the outcome in the snackbar («Ссылка скопирована» / «Не удалось скопировать ссылку») — never swallow a copy failure silently or Safari users get no link and no feedback. Caveat: the fallback runs synchronously only when the async API is *absent*; if it's *present but rejects*, the fallback runs after an `await` and WebKit may have consumed the user-gesture activation. Tests assign `document.execCommand`/`navigator.clipboard` directly (jsdom implements neither) rather than `vi.spyOn`.

- **Auth is cookie-session, no logout** — every `/api/*` route except `/api/auth/*` requires the signed `rmm_session` cookie (resolved to `req.user` by the `preHandler` in `app.ts`); data routes scope to `req.user.id` with ownership helpers in `server/src/ownership.ts` (cross-user id access → 404). `User.passwordHash` is **nullable** (`String?`) so `db push` can ADD COLUMN to the populated table; `ensureAdminUser` fills it on boot and seeds `synthmadness/6629`, inheriting the legacy `default-user` row's books. Nickname uniqueness/login is case-insensitive via raw `LOWER(name)` (Prisma SQLite has no `mode: "insensitive"`). Server tests that hit data routes must obtain a cookie via `signIn(app)` (`server/test/helpers.ts`) and inject it (the `inject(...)` wrapper in `api.test.ts`/`avatar.test.ts`); the service-direct suites stay anonymous because they create books under the preserved `default-user` (admin) id without touching the HTTP gate. `SESSION_SECRET` is set in `server/vitest.config.ts` so test cookies sign deterministically and no `session-secret` file is written.

- **API client & bodyless requests** — `web/src/api/client.ts` only sets `Content-Type: application/json` when a body is present. Fastify rejects an empty body that declares that content-type (`FST_ERR_CTP_EMPTY_JSON_BODY` → 400), which silently breaks `DELETE`s. Don't reintroduce a blanket content-type header.
- **Mind-map canvas updates** — `web/src/canvas/MindMap.tsx` re-initialises Cytoscape only when the *set* of node/edge ids changes (add/remove). Attribute-only edits (gender, name, age, role) are synced into the existing instance in place by a second effect that spreads all mutable `data` fields (so `label`, `avatar`, `avatarUri` propagate too). Both paths are needed; editing an existing node without the sync effect would not show until reload.
- **Canvas event handlers must read latest callbacks via refs** — because the init effect only re-runs on id-set changes, the `tap`/`dragfree` handlers bound there would otherwise capture a *stale* `onNodeTap`/`onNodeMoved` closure after an attribute-only edit. `MindMap.tsx` keeps `onNodeTapRef`/`onNodeMovedRef` updated on every render and the handlers call `*.current`. Symptom if broken: add an avatar, reopen the character without a page reload, and the modal shows the schematic (stale node had no `avatarUpdatedAt`) instead of the new avatar + «Изменить»/«Удалить» menu. Don't inline the prop directly into `cy.on(...)`. (`web/src/canvas/__tests__/MindMap.test.tsx` guards this — it forces Cytoscape's null renderer since jsdom has no 2d canvas.)
- **Canvas avatar URI encoding** — for the *schematic* branch, the avatar SVG contains `#` hex colours, so `graphAdapter.ts` wraps it with `encodeURIComponent` before the `data:image/svg+xml,` prefix. An unencoded `#` truncates the URI (Cytoscape reads it as a fragment) and the avatar silently vanishes — keep the encoding. (Custom avatars use the `GET` endpoint URL instead, so this only applies when `avatarUpdatedAt` is null.)
- **Schematic SVG needs explicit width/height for the canvas** — `avatarSvgMarkup` takes an opt-in `{ sized: true }` that adds `width="100" height="100"` (matching the `viewBox`). `graphAdapter.ts` sets it for the `data:image/svg+xml,` background-image; the React `Avatar` omits it on purpose (an outer `<svg>` defaults `width`/`height` to `100%`, so it fills its sized span). A viewBox-only SVG has *no resolvable intrinsic pixel size* as an image, so Cytoscape's `background-fit: cover` mis-positions the silhouette on browsers that don't fall back to the viewBox (seen on Chrome/Android — the avatar renders offset within the node). Keep the dimensions on the data-URI branch; don't add them to the inline path or it stops scaling to its span. (Custom avatars are raster WebP with real dimensions, so unaffected.)
- **Avatar cache-busting** — `GET /api/characters/:id/avatar` returns `Cache-Control: public, max-age=31536000, immutable`. That's safe *only because* the client always requests it with `?v=<avatarUpdatedAt>` (`api.avatarUrl`), and `CharacterAvatar.updatedAt` is `@updatedAt` so every upsert advances it; `BookScreen` re-fetches the graph after a save to pick up the new timestamp. Drop the `?v=` and an updated avatar will show stale forever.
- **Avatar upload body limit** — the `PUT` avatar route overrides Fastify's global 1 MB `bodyLimit` to 4 MB, because a ~2 MB image base64-encodes to ~2.7 MB of JSON. The real ceiling is the 2 MB *decoded* byte cap checked in the handler. Don't lower the route `bodyLimit` below the base64-expanded size of the byte cap.
- **`tsc` vs Vitest on whole-file edits** — Vitest runs via esbuild and silently ignores duplicate imports/declarations; `tsc` (the Docker/`npm run build` step) does not. When replacing a whole component file, verify there's exactly one import/interface block — a leftover duplicate passes tests but breaks the build with TS2300. Run `npx tsc --noEmit -p web/tsconfig.json` after large web edits.
- **Canvas layout spacing** — spacing constants live in `web/src/lib/layout.ts`. `MindMap.tsx` passes `nodeSpacing` (base × `SPACING_FACTOR = 5`) and a **per-edge** `edgeLength` callback (see "Connection-based canvas scaling") to the cola layout. Stored `posX`/`posY` live in the original `LAYOUT_BASELINE = 3` logical space; `graphAdapter.toElements` multiplies them by `POSITION_SCALE = SPACING_FACTOR / LAYOUT_BASELINE` (5/3) on load and `MindMap`'s `dragfree` handler divides by it before persisting. So existing hand-placed maps spread out with the factor too — keep the load/save scaling symmetric or positions drift on every drag.
- **Big-node overlap is prevented by label-aware overlap avoidance, not edge length** — `MindMap`'s cola layout sets `nodeDimensionsIncludeLabels: true` (+ explicit `avoidOverlap: true`) so overlap avoidance counts each node's avatar **and** label box, not just the avatar. Without it a highly-connected character's scaled name (`font 11 × scale`) sprawls over neighbours uncounted — `edgeLengthForNodes`' geometric floor only adds breathing room, it is *not* the guarantee. Verified against `cytoscape-cola`: it feeds `node.layoutDimensions(options)` (which honours `nodeDimensionsIncludeLabels`) into both the separation constraints and the per-node footprint, plus a `2 × nodeSpacing(50)` pad. Don't drop the flag thinking the edge length covers it.
- **Don't `await` a real animation frame in `MindMap` tests** — cola runs a continuous (infinite) layout; awaiting an animation frame lets it tick its descent in jsdom's null renderer, where label rectangles are degenerate (no text metrics) and webcola's overlap projection throws `Cannot read properties of null (reading 'next')` *after* the test, failing the run. The non-awaiting tests are safe because the test + RTL cleanup (`cy.destroy()` stops cola) finish before cola's first rAF tick fires. The fit-on-load test asserts `cy.fit` via a `Core.prototype` spy with **no** awaited frame (the initial fit is synchronous), so cola never ticks. Keep MindMap tests synchronous.
- **Server schema on boot** — there are no Prisma migrations; `server/src/server.ts` runs `prisma db push --skip-generate --accept-data-loss` at startup (idempotent). `prisma migrate deploy` is wrong here — it exits 0 without creating tables. `--accept-data-loss` is required because Prisma's data-loss guard fires **unconditionally** when ADDING a unique constraint (e.g. `@@unique([sourceId, targetId])`) — it can't tell the data is already deduped, so without the flag the deploy aborts with `Use the --accept-data-loss flag`. It's safe because `normalizeRelationships()` runs first (see below), so no rows are actually lost.
- **Docker** — npm workspaces hoist all deps to the **root** `node_modules` (there is no `server/node_modules`); the runtime image copies root `node_modules` and puts `/app/node_modules/.bin` on `PATH` so the `prisma` CLI is found under `node dist/server.js`. The server build needs `@types/node` (build is `tsc`, which dev/`tsx` and Vitest skip — so type errors in `server.ts` only surface in the Docker build).
- **Server tests** — `server/vitest.config.ts` sets `DATABASE_URL=file:./test.db` so the Prisma client and the schema-push agree on a throwaway DB. Without it, tests run against `dev.db` and fail on a fresh clone.
- **Undirected reconcile (incident edges + canonical order)** — `Relationship.color` is nullable (`null` = default `EDGE_COLOR`, never written). `reconcileRelationships(tx, bookId, characterId, connections)` looks at **both endpoints** (`sourceId == characterId` OR `targetId == characterId`), keying edges by `otherId`; it **creates missing edges in canonical order** (`min/max` of the pair so `sourceId < targetId`), deletes incident edges whose other end isn't desired, and **updates `role`/`color`** on changed rows (a connection re-roled "from the other side" updates the same row). Self-targets are ignored. Because both sides are inspected, an edge created from the other character is seen and never duplicated.
- **Relations wire shape is tested in two places** — the relations payload is now a **flat list of connections** (`{ otherId, role, color }`, validated by `relationConnectionSchema`), exercised by both `server/test/relationships.test.ts` (the service directly) **and** `server/test/api.test.ts` (end-to-end). When you change the shape, update *both* — a focused `relationships` run goes green while `api.test.ts` posts the old shape and 400s. Run the **full** `npm run test --workspace server` before declaring a schema/validation change done.
- **Empty role is `""`, never `NULL`** — `relationConnectionSchema.role` is `z.string().trim().max(30).optional().default("")`, so an omitted/blank role stores `""` (a bare, arrowless line). `@@unique([sourceId, targetId])` (with canonical storage) already forbids a second edge per pair, so role is a pure label. The «Роль» field in `RelationsModal` is marked «Необязательно».
- **Edges are undirected (no arrowhead)** — `MindMap.tsx` edge style sets `target-arrow-shape: "none"`; `line-color`/`label`/colour are unchanged. `graphAdapter.ts` still maps `sourceId`→`source`, `targetId`→`target` (Cytoscape needs both ends), but the rendered line has no direction. `web/src/canvas/__tests__/MindMap.test.tsx` asserts `style("target-arrow-shape") === "none"`.
- **Legacy normalization before `db push`** — `server/src/server.ts` calls `normalizeRelationships()` (`server/src/services/normalize.ts`) **before** `prisma db push` on boot. It's idempotent and guarded by "the `Relationship` table exists" (fresh DB → no-op): it collapses duplicate unordered pairs (keeping the **earliest** by `createdAt`) and canonicalises survivors (`sourceId < targetId`), so the new `@@unique([sourceId, targetId])` index can be created on existing volumes. The push itself still needs `--accept-data-loss` (see "Server schema on boot") — Prisma's guard fires on the constraint addition regardless of whether dupes remain. SQLite evaluates `UPDATE ... SET sourceId = targetId, targetId = sourceId` against pre-update values, so the swap is correct.
- **Title cap is 60, name cap is 30** — `server/src/schemas.ts` has two validators: `name30` (`min(1).max(30)`) for character `firstName`/`lastName`/`middleName` and the relationship `role` max, and `title60` (`min(1).max(60)`) for `bookCreateSchema`/`bookUpdateSchema`. Both book dialogs use `inputProps={{ maxLength: 60 }}`; keep the server `title60` and the client `maxLength: 60` in sync or an over-30 rename 400s silently (the dialog swallows the error). Optional name fields use `name30.optional().nullable()`; the web client sends `lastName.trim() || null` (mirrors `middleName`).
- **Relations modal picker is a `Popper`, not a `Popover`** — `RelationsModal.tsx` uses MUI `Popper` (non-modal) for the `@uiw/react-color` wheel, anchored to the swatch. A MUI `Popover` (nested modal) marks the parent `Dialog` `aria-hidden`, which makes the "Сохранить" button unreachable to `getByRole` while the picker is open. Keep it a `Popper` + `ClickAwayListener`.
- **Back-button closes modals, not routes** — overlays call `useBackClose(open, onClose)` (`web/src/lib/useBackClose.ts`), backed by the singleton `web/src/lib/backStack.ts`. Opening pushes a throwaway history sentinel **at the same URL** (state-marker only) so react-router doesn't navigate; a real `popstate` closes the top overlay; a programmatic close drops its sentinel via a guarded `history.go`. The `guardedPops` counter + microtask-batched `reconcile` swallow self-induced echoes and collapse simultaneous closes (e.g. delete-character closes ConfirmDialog *and* CharacterModal → one `history.go(-2)`). Don't replace it with naive per-hook `popstate` listeners — those cascade-close the parent. Tests must call `__resetBackStack()` and dispatch `popstate` manually (jsdom doesn't fire it from `history.go`). `AvatarCropDialog` ignores Back while `busy` (WebP baking).
- **Deceased overlay is one SVG, layered twice** — `deceasedOverlaySvg()` (`web/src/lib/avatarSvg.ts`) is the single source for the veil+X. The React `Avatar` draws it as an absolutely-positioned layer; `graphAdapter` emits it as `overlayUri` (a sized, `encodeURIComponent`'d `data:image/svg+xml,` URI, **`null` when alive** so `MindMap`'s in-place `data()` merge clears a stale overlay), which `MindMap`'s `background-image`/`background-fit` mappers layer over `avatarUri`. Don't re-inline the X geometry.
- **Nested MUI `Dialog` exit transition defeats sequential same-label clicks in tests** — `CommentsModal` opens `CommentEditDialog` (a `Dialog` over a `Dialog`); while the editor is open MUI marks the parent `aria-hidden`, so `getByRole("button", { name: /^сохранить$/i })` resolves to the *editor's* Save (this aria-hidden behaviour is relied upon, same root cause as the Popper/Popover note above). But after the editor's Save calls `setEditing(null)`, the editor stays mounted through its ~225ms exit transition, so a second immediate click on the same name **re-hits the editor's stale Save** and the parent modal's `onSave` never fires. In tests, wait for the editor to unmount between the two clicks: `await waitForElementToBeRemoved(() => screen.queryByRole("textbox"))` (the editor is the only thing rendering a textbox). Don't insert an arbitrary timer or weaken the assertion. (`web/src/components/__tests__/CommentsModal.test.tsx` guards this.)
- **`Comment.characterId` is unindexed** — `Comment` has no `@@index([characterId])` (unlike `CharacterAvatar`, whose `characterId` is its `@id`), so `reconcileComments` and the graph load table-scan the `Comment` rows. Harmless at current scale; add `@@index([characterId])` to `server/prisma/schema.prisma` if comment volume ever grows.

## Tooling rules

- **File navigation** — always use Serena MCP tools (`mcp__plugin_serena_serena__find_file`, `mcp__plugin_serena_serena__list_dir`, `mcp__plugin_serena_serena__find_symbol`, `mcp__plugin_serena_serena__search_for_pattern`) instead of `find`/`grep`/Bash for locating files and symbols.
- **Reading files** — prefer `mcp__plugin_serena_serena__read_file` over the Read tool or `cat`.
- **Editing code** — always use Serena MCP editing tools (`mcp__plugin_serena_serena__replace_symbol_body`, `mcp__plugin_serena_serena__insert_after_symbol`, `mcp__plugin_serena_serena__insert_before_symbol`, `mcp__plugin_serena_serena__replace_content`, `mcp__plugin_serena_serena__create_text_file`) instead of the Edit/Write tools.
- At session start, call `mcp__plugin_serena_serena__initial_instructions` before any coding task to load the Serena instructions manual.
- **Project memories** — Serena onboarding has been performed; read `mem:core` first (graph root linking `tech_stack`, `conventions`, `task_completion`, `notebooks`, etc.) via `mcp__plugin_serena_serena__read_memory` before deep work.

**Model selection:**
- Describing changes, planning, analysis, code review — use **Opus 4.8** (`claude-opus-4-8`) with **medium thinking effort**.
- Implementing changes (editing/writing files) — use **Sonnet 4.6** (`claude-sonnet-4-6`) or **Haiku 4.5** (`claude-haiku-4-5-20251001`) to minimise token spend.
